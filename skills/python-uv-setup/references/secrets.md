# Sharing env values with SOPS + age

Secrets travel the same pipe as code: encrypted env files are committed to
the repo, decrypted locally. Adding a variable is one reviewed PR; teammates
get the value via `git pull`, never via Slack. Most of the team is on
Windows — commands below are PowerShell first.

## What's committed vs. not

| Path | Committed? | What it is |
|------|------------|------------|
| `.sops.yaml` | yes | team's age public keys + which files they encrypt |
| `secrets/dev.env`, `secrets/staging.env`, `secrets/prod.env` | yes (encrypted) | source of truth for each environment's values |
| `.env.example` | yes | variable names + comments (documentation) |
| `.env.dev`, `.env.staging`, `.env` | NO (gitignored) | generated by decrypting; disposable |
| `keys.txt` (the age private key) | NEVER — lives outside the repo | one per developer |

`.env.*` files are now generated artifacts — regenerate freely, never edit
them directly; edit the encrypted source instead.

## One-time developer setup

1. Install both tools:

   ```powershell
   winget install FiloSottile.age Mozilla.SOPS
   ```

   (Alternatives: `scoop install age sops` / `choco install age.portable sops`.
   If the corporate proxy blocks installers, both are single static .exe files —
   download from the GitHub releases pages and put them on PATH.)
   macOS/Linux: `brew install age sops`.

2. Generate a keypair into the path sops checks by default on Windows:

   ```powershell
   mkdir -Force $env:APPDATA\sops\age
   age-keygen -o $env:APPDATA\sops\age\keys.txt
   ```

   macOS/Linux: `age-keygen -o ~/.config/sops/age/keys.txt`. If a machine
   uses a nonstandard path, point sops at it with the `SOPS_AGE_KEY_FILE`
   user env var.

3. The command prints your **public key** (`age1...`) — it's also
   recoverable any time with:

   ```powershell
   age-keygen -y $env:APPDATA\sops\age\keys.txt
   ```

   The public key is safe to share anywhere, including Slack — that's the
   point of it. The `keys.txt` file is the private key: never share it,
   never commit it, back it up in your password manager. 

4. Send the public key to a maintainer (or open the PR yourself): add it to
   `.sops.yaml`, then any EXISTING member re-encrypts and commits:

   ```powershell
   sops updatekeys secrets/dev.env
   sops updatekeys secrets/staging.env   # repeat per file; confirm y
   ```

   After that lands, the new dev pulls and can decrypt. (A new key can't
   decrypt history from before it was added — that's a feature.)

5. Set an editor for sops edits (once, in your PowerShell profile):

   ```powershell
   $env:EDITOR = "code --wait"
   ```

## Repo bootstrap (first time a repo adopts this)

`.sops.yaml` at the repo root:

```yaml
creation_rules:
  - path_regex: secrets/.*\.env$
    age: >-
      age1jacob...,
      age1dev2...
```

Encrypt the existing plaintext files — copy INTO `secrets/` first so the
creation rule matches, and verify before committing:

```powershell
mkdir -Force secrets
cp .env.dev secrets/dev.env
sops encrypt -i secrets/dev.env
Get-Content secrets/dev.env | Select-String ENC   # values must show ENC[AES256_GCM,...]
```

Only variable VALUES are encrypted — names stay readable, so PR diffs show
which keys changed. Keep prod decryptable by deploy tooling + two people,
not the whole team (use a separate creation rule for `secrets/prod.env`).

## Daily use

```powershell
# materialize a local env file (use --output; see gotcha below)
sops decrypt secrets/dev.env --output .env.dev

# edit values / add a variable (decrypts to $EDITOR, re-encrypts on save)
sops secrets/dev.env

# adding a var = one PR: code change + sops edit + .env.example line
```

**Activating an environment:** on macOS/Linux the skill's symlink works
(`ln -sf .env.dev .env`). On Windows, skip symlinks (they need Developer
Mode) — decrypt straight to `.env` instead; switching = decrypting a
different file:

```powershell
sops decrypt secrets/dev.env --output .env
```

Since `.env` is generated either way, nothing is lost by overwriting it.
Optional hardening for prod-adjacent work: `sops exec-env secrets/staging.env 'uv run serve'`
runs a command with the vars injected and never writes plaintext to disk.

## Gotchas

- **Never use `>` redirection in PowerShell** (`sops decrypt ... > .env`) —
  Windows PowerShell writes UTF-16 with a BOM and the resulting .env file
  breaks parsers in confusing ways. Always use `--output`.
- Don't `git add` a plaintext file into `secrets/` — encrypt in place first
  and eyeball for `ENC[` before committing. A `gitleaks` pre-commit/CI hook
  is cheap insurance.
- Lost laptop / leaked `keys.txt`: remove that public key from `.sops.yaml`,
  `sops updatekeys secrets/*.env`, and **rotate the underlying secrets** —
  re-encrypting alone doesn't un-leak old values (git history remains
  decryptable by the old key).
- Anything ever pasted in Slack is leaked — rotate it as it migrates into
  `secrets/`.
